Apache Fluss 列裁剪功能调研笔记

Posted on 2025-09-08 Edited on 2025-09-21 In 学习记录

最近因工作需要调研开源不久的Apache Fluss，初步看了它的核心特性后，列裁剪功能让我很感兴趣——毕竟大数据场景下读更少的数据是提升查询效率的关键。官方文档对这块的介绍比较简略，所以我结合源码和自定义Demo做了深入研究，整理成这份笔记。

一、调研背景与列裁剪的核心价值

在传统的流式消费场景中，如果某个topic有很多列，而客户端实际只需要一部分列的信息，往往都需要消费完整的数据，然后再客户端进行列筛选，如果需要的列占比比较少，那其实这会造成非常大的网络IO的浪费。

Fluss的列裁剪正好解决这个问题，核心价值有两点：

  1. 减少I/O开销：只读取需要的列，无关列完全跳过，尤其宽表场景下数据量减少非常明显；
  2. 无需解压即可裁剪：依赖Arrow列存的单列压缩特性，裁剪时不用解压整批数据，服务端处理效率大幅提升。

二、官方文档核心信息提炼

Fluss官方文档对列裁剪的描述集中在Log Table部分，核心信息就两点，我帮着拆解一下：

  1. 存储基础：数据默认用Apache Arrow列存格式，这是列裁剪的前提———列存才能精准定位某列数据，行存做不到；
  2. 压缩协同：Arrow格式对每一列单独压缩，裁剪时不需要解压整个数据块，直接跳过无关列的压缩块即可。

官方原文如下，方便对照：

In Fluss, Log Tables are stored in a columnar format by default (i.e., Apache Arrow). This format stores data column-by-column rather than row-by-row, making it highly efficient for column pruning. When a query specifies only a subset of columns, Fluss skips reading irrelevant columns entirely. This enables efficient column pruning during query execution and ensures that only the required columns are read, minimizing I/O overhead and improving overall system efficiency.

但文档只讲了是什么，没说怎么实现，所以必须挖源码。

3.2.1 准备阶段：setCurrentProjection() 方法

这是整个列裁剪的初始化，从代码里能看到它主要做3件核心事：

  1. 缓存复用：按tableId缓存ProjectionInfo（裁剪配置），避免重复生成；
  2. 嵌套字段扁平化：通过flattenFields()递归处理嵌套结构，把多维字段转成一维数组，这样裁剪时能精准定位子列；
  3. 裁剪位图生成：
    • nodesProjection：记录拍平后哪些字段要保留；
    • buffersProjection：Arrow中一列对应多个Buffer（如Bool列有validity buffer + data buffer），这个位图直接标记哪些Buffer要保留，后续筛选时只需要把对应的buffer取出来即可。

3.2.2 执行阶段：project() 与 projectArrowBatch() 方法

此阶段为实际数据裁剪的执行流程，具体步骤如下：

  1. 解析元数据：从Arrow数据块中读Message对象，拿到原始RecordBatch的字段和Buffer信息；
  2. 核心裁剪操作：projectArrowBatch()方法按位图筛选字段newNodes和BufferselectedBuffers，这里只记录Buffer的偏移量和长度，不做内存拷贝；
  3. 数据重构：把裁剪后的字段、Buffer重新封装成新的Log Batch，包含日志头、新的Arrow元数据和Buffer。

五、总结

Fluss列裁剪的实现逻辑清晰且高效，核心依赖两点：

  1. Arrow 列存储与单列压缩：保证“精准选列”和“无需解压”，是裁剪的基础；
  2. 缓存与位图筛选机制：通过缓存减少重复计算，位图快速定位Buffer，无内存拷贝，性能拉满。

在实际业务场景中，列裁剪功能可显著降低网络 I/O 开销，提升查询效率。但需注意，Fluss 的列裁剪操作在内存中执行，若缓存未命中需读取磁盘时，磁盘 I/O 仍为全量读取，这是该功能当前存在的局限。
